# PHASE 5: TRANSACTIONS (Day 9)
> CRITICAL FOR BANKING — NOT IN VIDEOS

## 5.1 @Transactional Basics

```java
@Transactional
public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
    Account from = accountRepository.findById(fromId).orElseThrow(...);
    Account to = accountRepository.findById(toId).orElseThrow(...);

    if (from.getBalance().compareTo(amount) < 0) {
        throw new BusinessException(ErrorCode.INSUFFICIENT_BALANCE);
    }

    from.setBalance(from.getBalance().subtract(amount));
    to.setBalance(to.getBalance().add(amount));
    // Auto-commit on method exit
    // Auto-rollback on RuntimeException
}
```

## 5.2 Rollback Rules

```java
@Transactional(
    rollbackFor = Exception.class,        // rollback on ALL exceptions (not just RuntimeException)
    noRollbackFor = BusinessException.class // don't rollback for this
)
```

- [ ] Default: rollback on `RuntimeException`, NOT on checked exceptions
- [ ] Always use `rollbackFor = Exception.class` in banking
- [ ] `noRollbackFor` for expected business exceptions

## 5.3 Propagation Types

| Propagation | Behavior |
|---|---|
| `REQUIRED` | Join existing or create new (default) |
| `REQUIRES_NEW` | Always create new, suspend existing |
| `NESTED` | Nested transaction with savepoint |
| `SUPPORTS` | Join if exists, run non-transactional if not |
| `NOT_SUPPORTED` | Always run non-transactional |
| `MANDATORY` | Must run inside existing transaction |
| `NEVER` | Must NOT run inside transaction |

```java
// Banking example: audit log should commit EVEN IF main transaction rolls back
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveAuditLog(AuditLog log) {
    auditLogRepository.save(log);
}
```

## 5.4 Isolation Levels

| Level | Dirty Read | Non-Repeatable Read | Phantom Read | Performance |
|---|---|---|---|---|
| `READ_UNCOMMITTED` | Yes | Yes | Yes | Fastest |
| `READ_COMMITTED` | No | Yes | Yes | Fast |
| `REPEATABLE_READ` | No | No | Yes | Medium |
| `SERIALIZABLE` | No | No | No | Slowest |

```java
@Transactional(isolation = Isolation.READ_COMMITTED)
public Account getAccount(Long id) { ... }
```

- [ ] When to use which isolation level
- [ ] PostgreSQL default: READ_COMMITTED
- [ ] Banking transfers: SERIALIZABLE or use pessimistic locking

## 6.1 Optimistic Locking

```java
@Entity
public class Account {
    @Id
    private Long id;

    @Version

    private BigDecimal balance;
}
```

```java
// When two threads try to update same account:
// Thread 1: reads version=1, updates -> version=2 (SUCCESS)
// Thread 2: reads version=1, updates -> version=1 != 2 (OptimisticLockException)
```

- [ ] Use when: low conflict, read-heavy
- [ ] Handle `OptimisticLockException` gracefully

## 6.2 Pessimistic Locking

```java
// Lock the row — nobody else can read/write until transaction completes
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a FROM Account a WHERE a.id = :id")
Optional<Account> findByIdForUpdate(@Param("id") Long id);
```

```java
@Transactional
public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
    // Lock both accounts — prevent concurrent modification
    Account from = accountRepository.findByIdForUpdate(fromId).orElseThrow(...);
    Account to = accountRepository.findByIdForUpdate(toId).orElseThrow(...);

    from.setBalance(from.getBalance().subtract(amount));
    to.setBalance(to.getBalance().add(amount));
}
```

- [ ] `PESSIMISTIC_WRITE` — exclusive lock (nobody can read/write)
- [ ] `PESSIMISTIC_READ` — shared lock (others can read, not write)
- [ ] Use when: high conflict — money transfers, inventory

## 7.8 Logout & Token Invalidation

```java
// Logout handler — invalidate refresh token
@Component
@RequiredArgsConstructor
public class CustomLogoutHandler implements LogoutHandler {
    private final TokenRepository tokenRepository;

    @Override
    public void logout(HttpServletRequest request, HttpServletResponse response, Authentication auth) {
        String token = jwtService.extractToken(request);
        if (token != null) {
            tokenRepository.findByToken(token).ifPresent(storedToken -> {
                storedToken.setRevoked(true);
                storedToken.setExpired(true);
                tokenRepository.save(storedToken);
            });
        }
    }
}
```

public class TransferEventConsumer {

    @RabbitListener(queues = "transfer.queue")
    public void handleTransfer(TransferEvent event) {
        // Check if already processed
        if (processedEventRepository.existsByEventId(event.getEventId())) {
            log.warn("Duplicate event detected: {}", event.getEventId());
            return;  // Skip — already processed
        }

        // Process transfer
        transferService.processTransfer(event);

        // Mark as processed
        processedEventRepository.save(new ProcessedEvent(event.getEventId()));
    }

## 14.1 Resilience4j

```xml
<dependency>
    <groupId>io.github.resilience4j</groupId>
    <artifactId>resilience4j-spring-boot3</artifactId>
</dependency>
```

```java
@Service
public class PaymentService {

    @CircuitBreaker(name = "paymentService", fallbackMethod = "processPaymentFallback")
    @Retry(name = "paymentService")
    @RateLimiter(name = "paymentService")
    @TimeLimiter(name = "paymentService")
    public CompletableFuture<PaymentResponse> processPayment(PaymentRequest request) {
        // Call external payment gateway
        return CompletableFuture.supplyAsync(() -> paymentClient.process(request));
    }

    private CompletableFuture<PaymentResponse> processPaymentFallback(PaymentRequest request, Throwable t) {
        log.error("Payment service fallback triggered", t);
        return CompletableFuture.completedFuture(new PaymentResponse("PENDING", "Service unavailable"));
    }
}
```

```yaml
resilience4j:
  circuitbreaker:
    instances:
      paymentService:
        sliding-window-size: 10
        failure-rate-threshold: 50
        wait-duration-in-open-state: 30s
        permitted-number-of-calls-in-half-open-state: 3
  retry:
    instances:
      paymentService:
        max-attempts: 3
        wait-duration: 1s
        retry-exceptions:
          - java.io.IOException
          - java.util.concurrent.TimeoutException
  rate-limiter:
    instances:
      paymentService:
        limit-for-period: 10
        limit-refresh-period: 1s
        timeout-duration: 0
```

| Pattern | Purpose |
|---|---|
| Circuit Breaker | Stop calling failing service |
| Retry | Retry on transient failure |
| Rate Limiter | Limit calls per time window |
| Time Limiter | Timeout for slow calls |
